# Relify 项目需求文档 (PRD)

## 1. 项目概述
**Relify** 是一个可扩展的多平台即时通讯消息桥接（Bridge）系统。其核心目标是打通不同聊天平台（如 Matrix, QQ 等）之间的消息壁垒，实现消息的双向或多向同步转发。系统支持灵活的路由拓扑（Hub 模式或 Peer 模式），并具备持久化存储和自动房间映射功能。

## 2. 系统架构与核心概念

### 2.1 平台适配器抽象 (Platform Adapter)
系统必须定义统一的接口规范，以屏蔽不同聊天平台的底层协议差异。
*   **核心能力**：必须支持发送消息、编辑消息、撤回消息、上传文件、创建房间、获取房间信息。
*   **路由类型 (Route Type)**：每个平台需定义其路由特征：
    *   **Mirror (镜像型)**：平台具备动态创建房间的能力，可为外部的一个房间创建一个对应的新房间（如 Matrix）。
    *   **Aggregate (聚合型)**：平台通常绑定到一个固定的群组/房间，不具备动态创建对应房间的能力，所有外部消息均转发至此单一房间（如 QQ 群）。
*   **虚拟用户 (Ghost User)**：支持在目标平台使用“傀儡”账号模仿源用户的昵称和头像（视平台能力而定）。

### 2.2 路由模式 (Routing Modes)
系统需支持两种互斥的路由拓扑结构，通过配置文件切换：
*   **Hub 模式 (星型拓扑)**：
    *   指定一个平台作为中心节点（Hub Platform，如 Matrix）。
    *   其他平台（Spoke）的所有消息仅转发至 Hub 平台。
    *   Hub 平台会自动为每个来源房间创建一个对应的“桥接房间”。
    *   适用于以某一平台为主力，聚合管理其他平台消息的场景。
*   **Peer 模式 (网状拓扑)**：
    *   无中心节点，所有启用的平台之间两两互联。
    *   当一个平台收到消息时，自动转发给除自己以外的所有其他启用平台。
    *   系统需根据目标平台的 `RouteType` 决定是创建新房间还是复用配置的聚合房间。

## 3. 功能详细需求

### 3.1 消息处理与转换
*   **统一消息模型**：系统内部使用统一的消息结构（Message），包含发送者信息、房间ID、时间戳等。
*   **消息分段 (Segmentation)**：支持富文本消息的解析与重组，支持以下类型：
    *   纯文本 (Text)
    *   图片 (Image) - 需携带 URL
    *   音频 (Audio) - 需携带 URL
    *   视频 (Video) - 需携带 URL
    *   文件 (File) - 需携带 URL
    *   提及/At (Mention)
*   **操作类型**：需支持以下事件操作的同步：
    *   **Create**：新消息发送。
    *   **Update**：消息编辑/修改（需映射原消息ID）。
    *   **Delete**：消息撤回/删除（需映射原消息ID）。
*   **引用回复 (Reply)**：必须处理消息的引用关系，将源平台的 `ReplyToID` 转换为目标平台的对应消息ID。

### 3.2 房间绑定与自动化
*   **动态绑定 (Dynamic Binding)**：
    *   当系统收到来自未绑定房间的消息时，需根据当前的路由模式自动创建绑定关系。
    *   在 **Hub 模式**下：自动在 Hub 平台创建新房间，并建立 `Source Room <-> Hub Room` 的绑定。
    *   在 **Peer 模式**下：自动在所有 Mirror 类型的 Peer 平台创建新房间；对于 Aggregate 类型的 Peer 平台，直接绑定其预设房间。
*   **绑定持久化**：一旦绑定建立，系统需将绑定关系（Binding ID, 平台, 房间ID, 配置）写入数据库，后续消息直接查找路由，不再重复创建。

### 3.3 消息防回响 (Echo Prevention)
*   系统必须识别并忽略由机器人自身发送的消息。
*   通过数据库记录的消息映射表，判断消息是否为桥接产生的回声，避免死循环。

## 4. 平台适配器需求

### 4.1 QQ 适配器
*   **协议**：基于 OneBot 标准协议。
*   **连接方式**：支持 WebSocket Client 模式连接 OneBot 实现端（如 LLOneBot, NapCat 等）。
*   **路由类型**：`RouteTypeAggregate`（聚合型）。
*   **配置项**：
    *   `protocol`: 协议类型（主要支持 ws）。
    *   `url`: WebSocket 地址。
    *   `secret`: 鉴权 Token。
    *   `group`: **必须配置**，指定唯一的桥接群号。
*   **特性**：
    *   不动态创建群，所有消息转发至配置的 `group`。
    *   发送消息时支持 Text, Image, Audio, Video。
    *   CreateRoom 接口仅返回配置的 `group` ID。

### 4.2 Matrix 适配器
*   **协议**：基于 Matrix Application Service (AS) API。
*   **连接方式**：作为 AppService 运行，需提供 HTTP 监听端口供 Homeserver 推送事件。
*   **路由类型**：`RouteTypeMirror`（镜像型）。
*   **配置项**：
    *   `homeserver_url`: Matrix 服务器地址。
    *   `domain`: 服务器域名。
    *   `as_token` / `hs_token`: AS 鉴权令牌。
    *   `bot_localpart`: 机器人账号前缀。
    *   `user_namespace`: 虚拟用户的前缀（如 `relify_`）。
    *   `auto_invite_user`: 创建房间后自动邀请的管理员用户ID。
*   **特性**：
    *   **虚拟用户**：需根据源平台的用户信息，动态注册/更新 Matrix 虚拟用户（Ghost User），设置昵称和头像。
    *   **富文本渲染**：支持将消息体渲染为 HTML 格式（支持引用块、图片标签、链接）。
    *   **媒体处理**：支持解析 mxc:// URI 并转换为 HTTP 下载链接。

## 5. 数据存储需求

*   **数据库选型**：使用 SQLite 嵌入式数据库，确保无需额外部署数据库服务。
*   **性能优化**：
    *   开启 WAL (Write-Ahead Logging) 模式。
    *   实现异步写入缓冲队列（Buffer），避免高并发下的锁冲突。
*   **存储内容**：
    *   **Bindings**：房间绑定关系表。
    *   **BindingRooms**：绑定关系下的具体平台与房间映射。
    *   **MessageMappings**：源平台消息ID与目标平台消息ID的映射表（用于编辑、撤回、回复）。
    *   **Events**：原始事件日志。
    *   **Users**：用户资料缓存（昵称、头像）。
*   **数据清理**：需具备定时任务（如每4小时），清理过期的消息映射和事件日志（如保留48小时），防止数据库无限膨胀。

## 6. 非功能性需求 (NFR)

### 6.1 配置管理
*   使用 YAML 格式配置文件。
*   首次启动若配置文件不存在，需自动生成包含默认值的配置文件。
*   需支持配置校验（Validate），确保必要字段（如 Hub 模式下的 Hub Platform）已填写。

### 6.2 健壮性与容错
*   **重试机制**：消息转发失败时，需进行有限次数的指数退避重试（如重试3次）。
*   **优雅停机**：接收到系统中断信号（SIGINT/SIGTERM）时，应停止接收新消息，刷新数据库缓冲区，关闭连接。
*   **日志记录**：使用结构化日志（slog），支持 Info/Debug 级别切换。

### 6.3 部署
*   程序应编译为单二进制文件。
*   运行时会在工作目录下自动创建 `data` 目录存放数据库文件。
