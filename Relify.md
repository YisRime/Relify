# Relify 产品需求文档

## 1. 核心架构与性能需求

### 1.1. 架构模式

系统支持两种架构模式，通过配置文件切换：

**1.1.1. 无中心对等架构（peer-to-peer）**
- 所有接入平台逻辑对等
- 消息可以在任意平台间传递
- 路由关系灵活配置

**1.1.2. 中心平台架构（hub）**
- 设定一个中心平台（如 Matrix），必须在配置文件中显式指定
- 所有其他平台与中心平台建立两两绑定关系
- 消息只在中心平台处理，不会从中心平台转发回其他平台
- 适用于需要统一消息管理的场景

### 1.2. 其他架构需求

1.2.1. **六边形架构**：核心层只处理逻辑分发与 ID 映射，不包含任何平台特有的 API 实现。
1.2.2. **内存级直通**：移除所有中间件消息队列，依赖内存函数调用。
1.2.3. **即发即弃**：核心层触发分发后立即响应 HTTP 200，不等待下游结果。
1.2.4. **全链路异步**：I/O 密集型操作（网络请求、数据库写）必须在独立 Goroutine 中执行，严禁阻塞主流程。

## 2. 统一消息模型需求
2.1. **标准化数据流**：基于 Matrix 标准裁剪。
2.2. **消息指纹**：消息在内存流转时，必须确立唯一性。
2.3. **负载分类**：Payload 分为 `m.text`, `m.image`, `m.file` 等，核心层透传内容，不修改语义。
2.4. **入站清洗**：入站驱动负责将平台私有格式（如 MarkdownV2, Custom Emojis）清洗为通用 HTML 或标准 Markdown。

## 3. 路由与 ID 翻译需求

### 3.1. 中心平台路由模式（hub 架构）

仅在 `architecture_mode: "hub"` 时适用。

**3.1.1. 中心平台设定**

必须在配置文件中显式指定中心平台

**3.1.2. 两两绑定原则**

- 假设有三个平台：Telegram、Discord、Matrix（中心平台）
- 系统自动建立：Telegram ↔ Matrix 和 Discord ↔ Matrix 的绑定关系
- **不会**建立 Telegram ↔ Discord 的直接绑定

**3.1.3. 消息流向**

- Telegram → Matrix：Telegram 的消息发送到 Matrix，在 Matrix 上处理
- Discord → Matrix：Discord 的消息发送到 Matrix，在 Matrix 上处理
- **重要**：消息只在中心平台（Matrix）处理，不会从 Matrix 转发回 Telegram 或 Discord
- 用户在 Matrix 上可以查看和管理来自所有平台的消息

**3.1.4. 平台路由类型（RouteType）**

RouteType 是平台级别的属性，**不在配置文件中设置**，而是由各平台驱动在代码中定义。

各平台驱动需要实现 `GetRouteType()` 方法来声明自己的路由类型：

- **Mirror（镜像）模式**：
  - 该平台上的每个房间（room）都会在中心平台上创建一个对应的独立绑定房间
  - 例如：Telegram 上的 Room-A 会对应 Matrix 上的 Room-A-TG
  - 例如：Telegram 上的 Room-B 会对应 Matrix 上的 Room-B-TG
  - 这是一对一的房间绑定关系

- **Aggregate（聚合）模式**：
  - 该平台上的多个房间可以聚合绑定到中心平台的同一个房间
  - 例如：Telegram 上的 Room-A、Room-B、Room-C 可以都绑定到 Matrix 上的 Room-TG-All
  - 这是多对一的房间绑定关系

### 3.2. 引用/回复的 ID 翻译 (ID Translation)

- **入站处理**：当入站平台驱动检测到消息是"回复消息"时，提取原始引用 ID (`RefSourceID`)。
- **核心映射**：核心层必须查询 **ID 映射表**，查找 `RefSourceID` 在目标平台对应的 `TargetMessageID`。
- **Payload 替换**：
  - **若找到映射**：将消息对象中的引用字段替换为目标的 `TargetMessageID`，下游平台驱动即可发起原生回复。
  - **若未找到映射**：核心层需将消息降级为普通消息，放弃引用处理。

### 3.3. 智能提及转换
核心层尝试将 `@UserA` 根据映射表转换为目标平台的 `<@UserA_ID>`，转换失败则保留纯文本。

## 4. 平台驱动实现与呈现逻辑
4.1. **统一身份分发**：
   - 核心层下发统一的 `SenderProfile`。
   - **出站平台驱动** 根据目标平台支持（Webhook/Puppet/Bot API）使用原生伪装、虚拟账号还是在消息头拼接 `[User: 昵称]`。
4.2. **内容确定性渲染**：
   - 平台驱动自行实现渲染富文本、剥离标签发送纯文本、还是对超长文本进行截断。
4.3. **非标消息处理**：红包、投票等类型，平台驱动无法原生支持时，必须渲染为 `[描述文本]`。
4.4. **状态同步（基于 ID 映射）**：
   - **编辑/撤回**：当收到编辑或撤回事件，平台驱动利用核心层提供的 `TargetMessageID` 调用对应 API。若无对应 ID 或 API 不支持，则忽略或发送补充提示。

## 5. 数据持久化需求
5.1. **嵌入式 SQLite (WAL模式)**：仅用于存储配置、绑定关系及 ID 映射日志。
5.2. **ID 映射日志 (Message Map Logs)**：
   - **结构**：必须存储 `(SourcePlatform, SourceMsgID)` 与 `(TargetPlatform, TargetMsgID)` 的多对多映射关系。
   - **写入流程**：出站平台驱动发送消息成功后，**异步**回调核心层写入该映射关系。
   - **生命周期**：设置 TTL（如 24-48 小时），过期自动清理。这直接决定了可回溯引用和编辑的时间窗口。
5.3. **路由表常驻内存**：绑定关系启动时全量加载，路由判定不查库。

## 6. 媒体与资源管理
6.1. **元数据流转**：核心层仅传递媒体 URL 和 Metadata，不触碰二进制流。
6.2. **资产代理 (Asset Proxy)**：针对私有链接，入站平台驱动生成内部代理 URL，出站平台驱动请求时流式转发。
6.3. **体积适配**：出站平台驱动检测体积超标时，自动转为"预览图+链接"模式。

## 7. 配置与扩展需求
7.1. **YAML 配置**：单文件或多文件配置管理。
7.2. **插件化平台驱动**：新增平台不修改核心代码。
7.3. **独立限流**：平台驱动内部维护 Token Bucket，触达限流时自行处理（排队或丢弃），不阻塞核心。
7.4. **可观测性**：结构化 JSON 日志（Level, Module, Msg）。
